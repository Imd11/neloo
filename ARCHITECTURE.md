六、完整数据流程图

┌─────────────────────────────────────────────────────────────────────────┐
│                         用户交互完整流程                                  │
└─────────────────────────────────────────────────────────────────────────┘

用户: "分析这个销售数据的趋势"
│
├─ Step 1: 文件上传
│  │
│  ┌──► Frontend: ChatInput 选择文件 (sales.csv)
│  │
│  ┌──► POST /files/upload
│  │    ├─ 验证 JWT Token
│  │    ├─ 保存到 Supabase Storage: /user123/sales.csv
│  │    ├─ 创建数据库记录: files 表
│  │    ├─ 关联到当前 thread: thread_files 表
│  │    └─ 返回: sandbox_path = /home/user/data/sales.csv
│  │
│  └──► Frontend: 显示文件上传成功
│
├─ Step 2: 发送消息
│  │
│  ┌──► Frontend: sendMessage("分析这个销售数据的趋势")
│  │
│  ┌──► LangGraph Client: POST /threads/{thread_id}/stream
│  │
│  └──► Backend: LangGraph Runtime 启动
│
├─ Step 3: Agent 规划任务
│  │
│  ┌──► TodoListMiddleware 创建任务:
│  │    ├─ "加载销售数据"
│  │    ├─ "探索性数据分析"
│  │    ├─ "趋势分析与可视化"
│  │    └─ "生成分析报告"
│  │
│  └──► SSE 流式推送 → Frontend 显示 Todo 列表
│
├─ Step 4: Agent 调用工具 - 加载数据
│  │
│  ┌──► 调用: execute_python
│  │    Code: 
│  │    ```python
│  │    import pandas as pd
│  │    df = pd.read_csv('/home/user/data/sales.csv')
│  │    print(df.head())
│  │    ```
│  │
│  ┌──► Sandbox Executor (E2B):
│  │    ├─ 同步文件: Supabase → E2B
│  │    ├─ 执行代码
│  │    ├─ 捕获 stdout
│  │    └─ 返回结果
│  │
│  └──► 返回给 Agent:
│       "成功加载 1000 行数据，包含 date、sales、region 列"
│
├─ Step 5: Agent 调用 SubAgent - 并行分析
│  │
│  ┌──► 调用: task (并行 2 个 SubAgent)
│  │    ├─ eda-analyst: 探索性数据分析
│  │    └─ viz-analyst: 时间序列可视化
│  │
│  ┌──► SubAgent 1 (eda-analyst):
│  │    ├─ 调用 execute_python
│  │    │  Code: df.describe(), 缺失值检查, 异常值检测
│  │    └─ 返回: "数据质量良好,无缺失值,销售额范围..."
│  │
│  ┌──► SubAgent 2 (viz-analyst):
│  │    ├─ 调用 execute_python

## 一、项目概览

**Data Analyst** 是一个基于 LangGraph 的专业级数据分析 AI 助手，特别针对统计学和经济学实证研究设计。项目采用前后端分离架构，具有：

- ✅ **完整的生产环境部署能力**
- ✅ **安全的代码执行沙箱环境**
- ✅ **专业的经济学知识库支持**
- ✅ **优秀的用户交互体验**

### 核心特性

- 🤖 **智能 Agent**：基于 LangGraph + DeepAgents 框架
- 🔒 **安全执行**：E2B Cloud / Docker / Local 三种沙箱模式
- 📚 **知识增强**：RAG 检索经济学方法论知识库
- 🎨 **多模态输出**：支持 Markdown、LaTeX 数学公式、图表可视化
- 🔄 **并行任务**：SubAgent 专业化分工
- 👥 **人机协同**：HITL (Human-in-the-Loop) 审批机制

### Agent 核心 (backend/src/agent/graph.py)

**架构特点：**
- 使用 `deepagents.create_deep_agent()` 构建
- 自动集成 4 大中间件：TodoList、Filesystem、SubAgent、HITL

   - 在沙箱中执行 Python 代码

## 十、关键技术亮点

### 1. 上下文工程 (Context Engineering)

**Manus 策略 - 双形式输出：**

```
大输出处理流程:
─────────────────
execute_python 输出 50,000 字符
         │
         ▼
检测: len(stdout) > 3000
         │
         ├─ 保存完整输出:
         │  /home/user/data/result_abc123.txt
         │
         ├─ 生成摘要 (前 500 字符)
         │
         └─ 返回给 LLM:
            - 摘要 (500 字符) ← Agent 看到
            - 文件路径引用

用户看到: 完整 50,000 字符 (前端不压缩)
Agent 消耗: 500 字符 (节省 49,500 tokens)
```

**优点：**
- 大幅节省 LLM Token 消耗
- 用户体验不受影响
- 可通过文件路径恢复完整数据

### 2. SubAgent 并行架构

```
主 Agent
   │
   ├─ task("eda-analyst", "分析变量 X")
   │   └─► 独立对话 → 独立工具调用 → 返回结果
   │
   ├─ task("stats-analyst", "检验假设")
   │   └─► 独立对话 → 独立工具调用 → 返回结果
   │
   └─ task("viz-analyst", "可视化分布")
       └─► 独立对话 → 独立工具调用 → 返回结果
```

**优点：**
- 🚀 **并行执行**：节省时间
- 🎯 **专业分工**：提高分析质量
- 🔒 **独立上下文**：不污染主对话

### 3. RAG 知识增强

```
Agent 准备分析 → 检索知识库 → 应用最佳实践
                     │
                     ▼
            ┌─────────────────┐
            │  Chroma Vector  │
            │   Database      │
            ├─────────────────┤
            │ - DID 方法论    │
            │ - IV 检验步骤   │
            │ - 代码模板      │
            │ - 常见错误      │
            └─────────────────┘
                     │
                     ▼
            Agent 生成高质量分析
            (减少 Hallucination)
```

**知识库内容：**
- **methods/**：经济学方法论（DID、IV、面板数据等）
- **checklists/**：分析检查清单
- **errors/**：常见错误与解决方案
- **code_templates/**：Python 代码模板
- **papers/**：学术论文 PDF

### 4. 沙箱隔离设计

**核心原则：**
- ✅ **变量不持久化**：每次执行独立，防止状态污染
- ✅ **路径重写**：虚拟路径 → 实际路径映射
- ✅ **自动文件同步**：执行前后自动同步存储
- ✅ **多级安全**：E2B 云端 > Docker 容器 > Local 快速

### 5. 文件关联系统

**多对多关联：**
- 文件可关联到多个会话
- 会话可包含多个文件
- 支持本地删除（解除关联）和全局删除（完全删除）

**智能分类：**
- `uploaded/`：用户上传
- `generated/`：AI 生成数据
- `chart/`：图表 PNG
- `code/`：Python 脚本

### 6. HITL 人机协同

**审批机制：**
```
Agent 准备执行敏感操作
         │
         ▼
触发 HITL 中断
         │
         ▼
前端显示审批界面
         │
    ┌────┴────┐
    │         │
  Approve   Reject
    │         │
    ▼         ▼
  继续执行  取消操作
```

**可配置中断点：**
- 文件删除
- 外部 API 调用
- 敏感数据访问

### 7. 多模态输出

**支持格式：**
- ✅ Markdown（GitHub 风格）
- ✅ LaTeX 数学公式（KaTeX 渲染）
- ✅ 代码高亮（多语言支持）

## 3) 文件在哪些系统间流动（总览）

```text
[用户浏览器]
  ① 本地选择 File 对象
  ② FormData 上传 HTTP
        |
        v
[后端 FastAPI: backend/src/api/webapp.py]
  ③ 接收 UploadFile
  ④ 写入存储(本地磁盘 或 Supabase Storage)
  ⑤ 写 DB: files
  ⑥ 写 DB: thread_files (把文件绑定到 thread)
        |
        v
[执行沙箱: backend/src/sandbox/*]
  ⑦ 同步/可见化(让执行环境能读到上传文件)
  ⑧ 代码执行产出 generated/chart/code 再回写存储+DB
        |
        v
[前端文件面板: frontend/src/app/components/FilePanel.tsx]
  ⑨ 按 thread 拉取 files 列表、展示、下载、解绑
```

可以把它理解成两条主链路：

- **上传链路（uploaded）**：浏览器 → 后端上传接口 → 存储 → DB 关联 thread →（可选）同步到沙箱
- **产物链路（generated/chart/code）**：沙箱生成 → 存储 → DB 关联 thread → 前端展示/下载

## 4) 浏览器侧：从点击 Upload 到“进入上传队列”

相关代码：

- `frontend/src/app/components/ChatInput.tsx`
- `frontend/src/app/hooks/useDataFileUpload.ts`
- `frontend/src/app/components/DataFileUpload.tsx`

### 4.1 UI 触发与本地队列（先看到文件 chip）

```text
User click "Upload Data"
  -> triggerFileSelect()
     -> <input type="file"> 打开系统文件选择器
        -> onChange(e)
           -> addFiles(FileList)  // 立刻进入本地队列 files[]
              - 校验后缀/大小/重复
              - 生成 DataFile{id,status:"pending"}
              - UI 显示 chip
```

在这一步，文件仍然只是浏览器内存里的 `File` 对象，没有离开用户机器。

### 4.2 threadId 的作用（绑定“属于哪个对话线程”）

上传请求会携带 `langgraph_thread_id`（前端 URL query param `threadId`），用于在后端把文件与某个对话线程绑定。否则 FilePanel 无法按 thread 展示“这次对话的文件”。

## 5) 浏览器 → 后端：真正上传发生在什么时候、传了什么

相关代码（上传请求发生处）：

- `frontend/src/app/hooks/useDataFileUpload.ts`：`uploadFiles()`

### 5.1 自动上传触发点（autoUpload）

当满足以下条件时，hook 内部会自动开始上传：

- `autoUpload=true`
- `threadId` 已存在
- 当前没有正在上传：`isUploading=false`
- 队列里存在 `status === "pending"` 的文件

```text
files[] 有 pending
  AND threadId != null
  AND autoUpload = true
  -> uploadFiles()
```

### 5.2 HTTP 请求形态（FormData + 线程绑定）

每个 pending 文件构造一次 `FormData` 并 POST：

```text
POST {apiUrl}/files/upload
Headers:
  Authorization: Bearer <token>   (有登录)
  或 X-User-Id: default           (无登录/本地)
Body (multipart/form-data):
  file = <binary>
  langgraph_thread_id = <threadId>
```

## 6) 后端接收：写存储、写 DB、把文件“挂到 thread 上”

相关代码（方向性）：

- `backend/src/api/webapp.py`：`POST /files/upload`
- `backend/src/storage/*`：存储与 DB 逻辑
- `backend/supabase/migrations/*`：表与策略（`files`、`thread_files` 等）

后端在 `/files/upload` 中通常做三层工作：

### 6.1 接收与校验

FastAPI 接收 `UploadFile`，并做文件类型/大小等校验。

### 6.2 写存储（两种模式）

```text
(模式A) Supabase Storage:
  UploadFile -> Supabase bucket -> storage_path

(模式B) 本地存储:
  UploadFile -> backend 本地目录 -> local_path
```

### 6.3 写数据库（files 与 thread_files）

概念上分两步：

1) 在 `files` 表插入一条记录（`file_type = "uploaded"`）
2) 若请求携带 `langgraph_thread_id`，则在 `thread_files` 表创建关联记录（把文件绑定到该 thread）

实体关系（简化）：

```text
users (user_id)
   |
   v
threads (id, langgraph_thread_id, user_id, ...)
   |
   |  (many-to-many via thread_files)
   v
thread_files (thread_id, file_id)
   ^
   |
files (id, file_type, storage_path, size, mime_type, ...)
```

## 7) 后端 → 执行沙箱：上传文件如何“出现在可执行环境里”

相关代码（方向性）：

- `backend/src/sandbox/file_sync.py`
- `backend/src/sandbox/executor.py`

目标：让 Agent 在执行 Python/R 分析时，能在沙箱内读到用户上传文件。

### 7.1 “文件同步/映射”

```text
存储中的 uploaded 文件
  -> 同步/映射到 sandbox 可见目录
  -> 得到 sandbox_path（对执行器可用的路径）
```

因此上传响应通常会返回两类路径：

- `storage_path`：存储系统中的位置
- `sandbox_path`：执行环境中可直接读取的路径（关键）

### 7.2 Agent/执行器如何使用

执行时，Agent 会引用上传文件的 `sandbox_path`，由执行器在沙箱环境读取并处理。

---

## 8) 沙箱产物回流：generated/chart/code 如何进入 FilePanel

产物文件（图、表、代码、数据等）通常走与 uploaded 类似的路径：

```text
sandbox 生成文件
  -> 写存储（Supabase 或本地）
  -> 写 DB: files(file_type="generated"/"chart"/"code")
  -> 关联 thread_files（绑定到当前 thread）
```

最终 FilePanel 展示的不仅是 uploaded，也包括 generated/chart/code。

---

## 9) 前端 FilePanel：展示的是“线程维度的 DB 权威列表”

相关代码：

- `frontend/src/app/components/FilePanel.tsx`
- 后端接口：`GET /api/threads/{langgraph_thread_id}/files`

FilePanel 的核心点是：它展示的是“DB 中绑定到该 thread 的文件”，而不是本地上传队列。

```text
FilePanel(threadId)
  -> fetch `${apiUrl}/api/threads/${threadId}/files`
  -> data.files -> dbFiles
  -> 按 file_type 分组显示
```

因此常见现象是：

- 刚选择文件时：ChatInput 旁边会立刻出现 chip（本地队列）
- FilePanel：需要后端上传完成 + DB 写入/关联完成后才会出现；必要时可点 `Refresh` 触发重新拉取

---

## 10) 下载与解绑：文件如何从 thread “移除”

### 10.1 下载（download）

FilePanel 下载一般走后端下载接口（例如 `/api/files/{fileId}/download` 或 `download_url`），后端从存储读取并返回 blob，前端触发浏览器下载。

### 10.2 解绑（unlink from thread）

接口形态（方向性）：`DELETE /api/threads/{threadId}/files/{fileId}`

语义通常是“移除 thread 与 file 的关联”（删 `thread_files`），不一定删除存储对象本体（是否级联删除取决于后端策略）。
