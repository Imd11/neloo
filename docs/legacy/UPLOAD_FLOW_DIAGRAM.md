# 文件上传流程图

## 完整流程对比

### 传统方式（单阶段） ❌

```
用户选择文件
    ↓
等待输入消息
    ↓
点击发送
    ↓
创建 thread
    ↓
开始上传文件 ← 用户需要等待
    ↓
文件上传完成
    ↓
创建 files 记录
    ↓
完成

问题：
- 用户体验差：需要等待文件上传完成
- 无法预上传文件
- 上传失败会中断整个流程
```

### 两阶段方式（当前实现） ✅

```
阶段 1: 预上传（异步）
┌─────────────────────────┐
│ 用户选择文件             │
└─────────────────────────┘
            ↓
┌─────────────────────────┐
│ 立即上传到 Storage       │
│ 创建 upload_sessions     │
│ status = 'uploaded'      │
│ committed = false        │
└─────────────────────────┘
            ↓
┌─────────────────────────┐
│ 前端显示"已上传"         │
│ 用户可以继续操作         │
└─────────────────────────┘

        (用户输入消息...)

阶段 2: 提交（关联）
┌─────────────────────────┐
│ 用户点击发送             │
└─────────────────────────┘
            ↓
┌─────────────────────────┐
│ 创建/获取 thread_id      │
└─────────────────────────┘
            ↓
┌─────────────────────────┐
│ commitFiles(threadId)    │
│ - 更新 upload_sessions   │
│   committed = true       │
│ - 创建 files 记录        │
│ - 创建 thread_files 关联 │
└─────────────────────────┘
            ↓
┌─────────────────────────┐
│ 文件永久关联到对话       │
└─────────────────────────┘

优势：
- ✅ 即时反馈：文件立即上传
- ✅ 异步处理：不阻塞用户输入
- ✅ 可撤销：提交前可以删除
```

## 错误处理流程

### 场景 1: 上传失败

```
用户选择文件
    ↓
开始上传
    ↓
网络错误 ❌
    ↓
upload_sessions.status = 'error'
    ↓
前端显示错误提示
    ↓
用户可以重试上传
```

### 场景 2: 提交失败

```
文件已上传（status = 'uploaded'）
    ↓
用户点击发送
    ↓
commitFiles() 调用失败 ❌
    ↓
upload_sessions 保持 committed = false
    ↓
前端显示错误提示
    ↓
用户可以重新发送（会再次尝试 commit）
```

### 场景 3: 文件过期

```
14:00 - 文件上传（expires_at = 15:00）
14:30 - 用户离开页面
15:01 - 后台清理任务运行
    ↓
检查：committed = false && expires_at < NOW()
    ↓
删除 upload_sessions 记录
    ↓
删除 Storage 文件
    ↓
清理完成 ✅
```

## 清理机制

### 定期清理任务

```python
# 每小时运行一次
async def cleanup_expired_uploads():
    """
    清理流程：
    1. 查找过期且未提交的记录
    2. 删除 Storage 文件
    3. 删除数据库记录
    """

    # 查找条件
    WHERE committed = false
      AND expires_at < NOW()

    # 影响的数据
    - upload_sessions 记录被删除
    - Storage 文件被删除
    - 用户界面中移除

    # 不影响的数据
    - 已提交的文件（committed = true）
    - files 表中的记录
    - thread_files 关联
```

### 清理时机

```
创建时间: 14:00
过期时间: 15:00 (TTL = 1小时)
清理时间: 15:00+ (下次清理任务运行)

┌────────┬────────┬────────┐
│ 14:00  │ 15:00  │ 15:30  │
├────────┼────────┼────────┤
│ 上传   │ 过期   │ 清理   │
└────────┴────────┴────────┘
```

## 关键设计点

### 1. 为什么 TTL 是 1 小时？

```
太短（如 5 分钟）：
❌ 用户可能还在思考问题
❌ 体验不好

太长（如 24 小时）：
❌ 占用存储空间
❌ 增加清理负担

1 小时刚好：
✅ 足够用户完成一次完整操作
✅ 不会浪费太多存储空间
```

### 2. 为什么需要 expected_size 和 actual_size？

```
expected_size: 前端告诉后端"我要上传多大的文件"
actual_size:   后端实际接收到的文件大小

验证：
if actual_size != expected_size:
    raise Error("文件上传不完整")

防止：
- 网络中断导致部分上传
- 恶意上传超大文件
```

### 3. 为什么 committed 和 status 都有？

```
committed: 布尔值，业务层面的状态
- false: 未关联到 thread
- true:  已关联到 thread

status: 字符串，技术层面的状态
- pending:   准备上传
- uploaded:  上传完成
- committed: 已提交
- error:     出错

组合使用：
- status='uploaded' + committed=false → 等待提交
- status='committed' + committed=true → 已永久保存
```

## 总结

**upload_sessions 实现了一个优雅的暂存机制**：

| 特性 | 说明 |
|------|------|
| 🚀 即时上传 | 文件立即上传，不等待对话创建 |
| ⏰ 自动清理 | 1小时后自动删除未提交的文件 |
| 🔄 可撤销 | 提交前可以删除文件 |
| 🛡️ 安全性 | RLS 策略保护用户数据 |
| 📊 状态跟踪 | 完整的状态机管理文件生命周期 |
| ♻️ 原子操作 | 事务保证数据一致性 |

这是一个经典的 **Two-Phase Commit** 模式在文件上传场景的应用。
